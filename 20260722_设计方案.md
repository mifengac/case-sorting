# 笔录转文本工具（case-sorting）设计方案

## 一、需求

- 把扫描的**打印体笔录**（PDF、图片）转成纯文本。
- 部署环境：内网**离线** Ubuntu 22.04 服务器，**纯 CPU，无 GPU**。
- 现有方案 Stirling-PDF（Java + 一堆系统依赖）和 PaddleOCR（要带 paddlepaddle 框架，几百 MB）都太重，要一个轻量的。

## 二、模型选型

查过 HuggingFace 和社区现状（2025-10 至 2026-07），候选如下：

| 方案 | 大小 | CPU 可用 | 打印体中文 | 手写中文 | 结论 |
|---|---|---|---|---|---|
| **RapidOCR + PP-OCRv5 mobile（ONNX）** | 模型共约 15~20 MB | ✅ 很快 | ✅ 很好 | 一般 | **选用** |
| PP-OCRv5 server（ONNX） | 约 100 MB | ✅ 稍慢 | ✅ 更好 | 较好 | 备用升级项 |
| PaddleOCR 本体 | 框架 500 MB+ | ✅ | ✅ | ✅ | 太重，放弃 |
| PaddleOCR-VL / DeepSeek-OCR / Chandra | 0.9B~9B | ❌ 需 GPU | 顶级 | 顶级 | 无 GPU 用不了，放弃 |

**最终决定：RapidOCR + PP-OCRv5 mobile 系列 ONNX 模型。**

理由（说人话版）：

- RapidOCR 是个开源的轻量推理库，它把 PaddleOCR 的模型转成了 ONNX 格式，跑模型只需要 `onnxruntime` 这一个约 50 MB 的 Python 包，**不用装 paddlepaddle 那个大框架**——模型效果和 PaddleOCR 一样，体积和依赖却小得多。
- PP-OCRv5 是百度 2025 年发的新一代模型，对简体中文、繁体、英文混排，以及竖排、生僻字都有升级，比 v4 端到端提升约 13 个百分点。mobile 版是给手机/CPU 用的小模型，打印体完全够用。
- 万一以后发现某些笔录有手写部分识别不好，可以把识别模型从 mobile 换成 server 版（还是 ONNX，约 100 MB），代码不用改，只换模型文件。这是留好的升级口。

三个模型文件（统一托管在魔搭社区 RapidAI/RapidOCR 官方仓库，`scripts/download_offline_bundle.sh` 会自动下载）：

- `PP-OCRv5_mobile_det`（文字检测：找出字在哪）约 5 MB
- `PP_OCRv4_cls`（方向分类：判断文字有没有倒）约 1.5 MB
- `PP-OCRv5_mobile_rec`（文字识别：读出是什么字）约 16 MB

## 三、整体架构

很简单，一条流水线：

```
上传 PDF/图片
    │
    ├─ PDF → PyMuPDF 逐页渲染成 300dpi 图片（纯 Python 包，无需系统依赖，离线友好）
    │
    ▼
图片预处理（可选：灰度化、自动纠偏）
    │
    ▼
RapidOCR 识别（det + cls + rec 三个 ONNX 模型，onnxruntime 跑 CPU）
    │
    ▼
文本后处理：按行坐标从上往下、从左往右排序拼接成段落
    │
    ▼
输出：页面预览 + 全文文本，可下载 .txt / .md
```

提供两种用法：

1. **网页服务**：FastAPI 起一个简单页面，拖拽上传 PDF/图片 → 后台识别 → 显示结果 → 复制/下载 txt。和团队其他项目（FastAPI 风格）保持一致。
2. **命令行批处理**：`python -m app.batch /path/to/目录` 把一整个目录的笔录批量转文本，结果存到输出目录，方便一次性处理存量档案。

## 四、技术栈与依赖

全部是纯 pip 包，无任何系统级依赖（不用 apt 装东西）：

- Python 3.10+（Ubuntu 22 自带 3.10）
- `rapidocr_onnxruntime`（自带 onnxruntime）
- `PyMuPDF`（PDF 转图片，自带渲染引擎，不需要 poppler）
- `fastapi` + `uvicorn`（网页服务）
- `Pillow`（图片处理）

总安装体积约 100 MB 出头（含模型），对比 Stirling-PDF 的 Java 运行时 + LibreOffice 等依赖，轻了一个数量级。

## 五、离线部署流程

内网不能联网，所以要在**有网的机器**上把所有东西打包，再拷进去：

1. 有网机器（最好也是 Ubuntu 22 + 同版本 Python）执行：
   - `pip download rapidocr_onnxruntime PyMuPDF fastapi uvicorn pillow -d wheels/` 把所有 whl 包下载到 `wheels/` 目录
   - 下载 3 个 ONNX 模型文件放到 `models/` 目录
2. 把项目代码 + `wheels/` + `models/` 一起拷进内网服务器。
3. 内网服务器执行：
   - `python3 -m venv .venv`
   - `.venv/bin/pip install --no-index --find-links=wheels rapidocr_onnxruntime PyMuPDF fastapi uvicorn pillow`
   - `.venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000`

项目里要附一个 `scripts/download_offline_bundle.sh`（有网机器上用）和一个 `scripts/install_offline.sh`（内网机器上用），把上面的步骤固化下来。

## 六、性能预估

- A4 纸 300dpi 一页，8 核 CPU 大约 2~5 秒。
- 一份 10 页笔录约半分钟内转完，可接受。
- 网页服务用简单的内存队列串行处理识别任务即可，不搞复杂的任务队列。

## 七、验收标准

- 上传一份扫描版打印笔录 PDF，能转出可读的 txt，段落顺序正确，错别字率在可人工校对范围内。
- 离线环境（拔网线）能完成安装和识别全流程。
- 批量模式能处理一个目录下的混合 PDF 和图片。
